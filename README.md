# 🌙 DreamDrift — سُكون

رفيق ليلي لتتبّع النوم والمزاج، مع تمارين تنفّس، مازج أصوات هادئة، واسترخاء موجّه بالصوت. تطبيق ويب تقدّمي (PWA) بملف واحد — بلا خادم خلفي، بلا بناء (build step)، كل البيانات تبقى على جهاز المستخدم.

**الإصدار الحالي:** 1.3.0

## ✨ المزايا

- تتبّع المزاج ونمط النوم اليومي
- تمارين تنفّس موجّهة (٤-٤-٤) بصوت مرافق
- مازج أصوات (مطر، أمواج، غابة، نار، رياح، ضوضاء بنية) مبني بالكامل بـ Web Audio API — بلا ملفات صوتية خارجية
- استرخاء موجّه بالصوت (مسح جسدي) عبر `speechSynthesis`، خلف بوابة جودة تتحقق من توفّر صوت مناسب
- **التحليل التفصيلي**: أربعة "وكلاء" تحليليون يقرؤون سجلات النوم — المِصفاة (تدقيق البيانات) ← البوصلة (تشخيص النمط) ← الفنار (فرز السلامة) ← المرساة (خطوة أسبوعية واحدة)
- إشعارات محلية لتذكير الاسترخاء ومنبّه الاستيقاظ عبر Service Worker (بلا خادم push)
- عمل كامل دون إنترنت بعد أول زيارة
- عربي/إنجليزي، وضع داكن/فاتح

│   ├── sleep-analysis.test.mjs    # وكلاء التحليل التفصيلي (المِصفاة/البوصلة/الفنار/المرساة)

## 🧪 الاختبارات

```bash
npm test
```

٧١ اختباراً عبر `node:test` المدمج في Node.js (بلا Jest أو Vitest أو أي حزمة خارجية) — يتطلب Node 20+. الاختبارات تستخرج الكود **الحقيقي** من `index.html` و`sw.js` وتُشغّله داخل بيئة `vm` معزولة بمحاكاة دنيا لعناصر DOM/`localStorage`/`speechSynthesis`/`IndexedDB`، بدل نسخ المنطق في ملفات منفصلة قد تنحرف عن الأصل.

تغطّي الاختبارات: تشخيص أنماط النوم، ترحيل البيانات عبر الإصدارات، بوابة جودة الصوت المرشد، ترتيب استئناف `AudioContext` قبل بناء العقد الصوتية، ومنطق جدولة/إطلاق/عدم تكرار إشعارات `sw.js`.

## 🔁 CI/CD

سير عمل واحد (`.github/workflows/ci.yml`) بمهمتين:

1. **test** — يعمل على كل push وكل pull request: يثبّت Node، يشغّل الاختبارات الـ71، ويتحقق من وجود ملفات PWA الأساسية.
2. **deploy** — يعمل فقط بعد نجاح `test`، وفقط عند push مباشر لفرع `main` (لا على pull requests): يرفع الملفات وينشرها على GitHub Pages عبر `actions/upload-pages-artifact` و`actions/deploy-pages` الرسميين.

**إعداد مطلوب مرة واحدة:** من إعدادات المستودع **Settings → Pages → Source**، اختر **GitHub Actions** بدل "Deploy from a branch". بدون هذا التغيير، مهمة `deploy` ستعمل لكن GitHub Pages لن يقبل نشرها.

## 🔔 كيف تعمل الإشعارات

المتصفح لا يسمح بمنبّهات حقيقية بلا خادم Push، لذا يعتمد `sw.js` على نمط شائع لتطبيقات PWA الشخصية:

- عند فتح التطبيق، تُرسِل الصفحة رسالة `CHECK_NOTIFICATIONS_NOW` لـ Service Worker.
- إن دعم المتصفح `periodicSync` ومُنح الإذن، يُفحص الجدول كل ١٥ دقيقة تلقائياً.
- الفحص يقارن الوقت الحالي بالموعد المحفوظ (نافذة ٥ دقائق) عبر IndexedDB، ولا يُطلق الإشعار أكثر من مرة في اليوم.

### 🩺 لوحة تشخيص مدمجة

من **الإعدادات → الإشعارات → تشخيص حالة الإشعارات** تظهر الحالة الفعلية على الجهاز: هل الإذن ممنوح، هل `periodicSync` مدعوم/ممنوح، الجدول المحفوظ فعلياً، وآخر مرة أُطلق فيها كل إشعار. هذه أول خطوة عند الإبلاغ عن مشكلة إشعارات — القيم الفعلية أدق من أي تخمين.

### ⚠️ قيود منصّة معروفة (لا عيوب تصميم)

1. **الإذن لم يُمنَح** — التطبيق لا يطلب الإذن تلقائياً أبداً (لتفادي رفض متاجر التطبيقات ومضايقة المستخدمين)؛ يجب الضغط صراحةً على مفتاح "تذكير التهدئة والاستيقاظ" في الإعدادات. بدونه، الجدول فارغ ولا شيء يُطلَق.
2. **`periodicSync` غير مضمون** — على Android/Chrome، يمنح Chrome إذنه تلقائياً بناءً على مدى استخدامك الفعلي للتطبيق المثبَّت (site engagement)، لا عبر نافذة طلب صريحة، وقد يستغرق منحه أياماً من الاستخدام المتكرر بعد التثبيت. **على iOS Safari، هذه الواجهة غير مدعومة إطلاقاً.**
3. **بالتالي على iOS تحديداً**: الإشعار لن يصل إلا إن كان التطبيق نفسه مفتوحاً (أو أُعيد فتحه) ضمن نافذة الخمس دقائق بعد الموعد — لا يوجد بديل بلا خادم Push حقيقي.
4. **وضع الحفظ (Doze) على أندرويد** قد يؤخّر `periodicSync` لساعات حتى مع منح الإذن.
5. **التثبيت من الشاشة الرئيسية مطلوب على iOS** لتفعيل أي سلوك PWA/إشعارات حقيقي؛ فتح الرابط داخل تبويب Safari عادي لا يكفي.

## 🚀 النشر على GitHub Pages

بعد إعداد المصدر أعلاه، أي push لفرع `main` يجتاز الاختبارات يُنشَر تلقائياً على:
`https://<username>.github.io/<repo-name>/`

> **ملاحظة PWA مهمة:** `manifest.json` و`sw.js` يُشيران إلى مسارات نسبية (`./`)، لذا يعملان بشكل صحيح سواء نُشر التطبيق في جذر الدومين أو داخل مسار فرعي (`/repo-name/`) كما هو حال GitHub Pages الافتراضي. لا حاجة لتعديل أي شيء.

## 🧪 التطوير محلياً

ملف واحد، لكن لتفعيل Service Worker بشكل صحيح يجب تقديمه عبر HTTP (لا `file://`):

```bash
python3 -m http.server 8080
# ثم افتح http://localhost:8080
```

## 📄 بنية البيانات

تُحفظ كل بيانات المستخدم محلياً عبر `localStorage` بنظام إصدارات (`SCHEMA_VERSION`) مع ترحيل تلقائي عند تغيّر البنية بين التحديثات. الإصدار الحالي: `4`.
